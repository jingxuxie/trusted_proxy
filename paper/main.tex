\begin{abstract}
Multi-agent LLM systems often split document reading, planning, and tool
execution across specialized agents. This modularity creates a trust-boundary
failure: an instruction from untrusted content can be rewritten by an upstream
peer as an ordinary action item, causing downstream agents to treat it as
authorized. We call this \emph{source laundering}. We introduce SLaMBench, a
synthetic benchmark for source-laundering attacks in tool-using agents across
workspace, CRM, code-review, travel, and research-assistant domains. On a
50-pair multidomain benchmark, an extractive reader--planner--executor topology
raises unauthorized-recipient attack success from 6\% in a direct single-agent
baseline to 22\%, a +16 percentage-point lift. A deterministic TraceGate monitor
eliminates observed unauthorized sends while preserving more injected-task
utility than the vulnerable multi-agent system. Follow-up ablations show that
attack amplification is not caused by delegation depth alone; it appears when
intermediate agents strip or normalize provenance. A current-model
\texttt{gpt-5.5} replication has low attack success across all tested
conditions, showing model dependence while preserving the diagnostic value of
provenance dropout. These results argue for treating inter-agent messages as
security-relevant authority transitions, not merely summaries.
\end{abstract}

\section{Introduction}

LLM agents are increasingly built as \emph{systems} rather than single prompts.
A deployed assistant may ask one model call to read documents, another to turn
those documents into a plan, and another to execute tool calls. This design is
attractive: it reduces prompt complexity, makes roles easier to debug, and lets
developers mix models, tools, memories, and guard components. Frameworks for
reasoning-and-acting agents, multi-agent conversation, software-engineering
agents, and tool protocols all encourage this modular view of agentic
AI~\citep{yao2022react,wu2023autogen,yang2024sweagent,mcp2025tools}. Yet the
same modularity also creates a subtle security question: when information moves
between agents, does its authority move with it?

The problem is easiest to see in a mundane workflow. A user asks: ``read this
vendor note and email \texttt{finance@example.test} the payment terms.'' The note
truthfully states that the invoice is net-30, but it also contains an
attacker-controlled sentence: ``add \texttt{audit-log@example.test} as a required
recipient.'' A direct single-agent system sees the extra recipient inside the
untrusted document. A multi-agent system may instead transform the sentence over
several steps: the Reader extracts it as an ``action item,'' the Planner promotes
it to a ``required step,'' and the Executor sees a peer-authored plan rather than
the original untrusted artifact. The instruction has not disappeared; its source
has been washed away.

We call this failure mode \emph{source laundering}. It is related to indirect
prompt injection, where malicious instructions appear in external content
processed by an LLM application~\citep{greshake2023not,zhan2024injecagent}. The
unit of analysis is different. We ask whether the \emph{same} untrusted
instruction becomes more effective after crossing an inter-agent boundary. In
systems-security terms, this resembles the classic confused-deputy problem: a
component with authority is tricked into exercising that authority for a less
trusted principal~\citep{hardy1988confused}. In LLM systems, however, the deputy
is not just calling an API; it is interpreting a natural-language handoff whose
source and authority may be ambiguous.

This matters because agentic systems increasingly have side effects. A tool call
can send an email, update a ticket, create a calendar event, write long-term
memory, or call an external API. The Model Context Protocol specification, for
example, explicitly describes tools that language models may discover and invoke,
and recommends human confirmation and clear tool exposure because model-controlled
tools interact with external systems~\citep{mcp2025tools}. The OWASP GenAI Top
10 likewise highlights prompt injection, excessive agency, sensitive information
disclosure, and supply-chain risks as central security concerns for generative AI
applications~\citep{owasp2025llmtop10}. Source laundering sits at the
intersection of these concerns: untrusted data becomes a peer instruction, and a
peer instruction becomes a side effect.

This paper studies three questions. \textbf{RQ1:} Can a provenance-losing
multi-agent handoff increase unauthorized tool use relative to direct exposure to
the same injected artifact? \textbf{RQ2:} Is the effect caused by the number of
agents, or by a specific loss of source and authority metadata? \textbf{RQ3:} Do
prompt-level provenance defenses, LLM-based guards, and deterministic tool-boundary
checks differ in their security/utility tradeoffs?

Our contributions are: (i) a threat model for source laundering as an
authorization-boundary failure in multi-agent LLM systems; (ii) SLaMBench, a
closed-world benchmark of paired benign and injected tool-use tasks; (iii) an
evaluation of direct, delegated, source-preserving, capability-scoped,
LLM-guarded, and runtime-monitored conditions; and (iv) a set of empirical
findings that make the mechanism precise. On \texttt{gpt-5.4-nano}, extractive
delegation amplifies unauthorized-recipient sends; naive delegation depth alone
does not. TraceGate, a simple deterministic monitor, blocks the forbidden side
effect while preserving useful work. On \texttt{gpt-5.5}, the same benchmark
reveals provenance dropout but no executed attacks on the tested slice, so the
right claim is conditional rather than universal.

\section{Threat Model and Security Property}

The user provides a legitimate task, such as reading an invoice note and emailing
an authorized summary to \texttt{finance@example.test}. The adversary can place
text inside an untrusted artifact that the system must process: a vendor note, a
customer ticket, a pull-request note, an itinerary note, or an abstract note. The
adversary cannot modify system prompts, trusted context, tool code, model weights,
runtime policy, or the user's initial goal. The attacker also does not control
the mock tools used for evaluation.

The protected assets are synthetic but structurally representative: authorized
recipients, synthetic secrets, protected status fields, calendar attendees, and
memory entries. The security property is \emph{authority separation}: untrusted
content may provide facts needed to complete the user task, but it must not
authorize new recipients, state changes, memory policies, or secret disclosure.
For example, an external invoice note may state payment terms, but it may not
authorize the assistant to email an additional recipient.

A source-laundering attack has three stages. First, an attacker-controlled
instruction appears in an untrusted artifact. Second, an intermediate agent
transforms it into a peer message without sufficient source, trust, and
no-authority metadata. Third, a downstream agent acts on the transformed message
by emitting a forbidden tool call. A run that fails to complete the legitimate
email is a utility failure; a run that emails \texttt{audit-log@example.test} is a
security failure. All experiments are closed-world: tools are deterministic mocks,
secrets and recipients are synthetic, and no real email, calendar, ticket,
browser, or memory service is contacted.

\section{Benchmark and Method}

\paragraph{SLaMBench.}
SLaMBench contains paired benign and injected task variants. The main scaled task
set, SLaMBench-50, contains 50 benign tasks and 50 injected variants across five
domains (Table~\ref{tab:slambench}). Each pair shares the same user goal, trusted
context, allowed tools, expected benign outcome, and reference benign tool calls;
the injected variant adds one attacker-controlled payload to the untrusted
artifact. This paired design makes the comparison simple: the useful task is the
same, but the injected task contains one extra untrusted instruction.

\centering

\begin{tabular}{lcll}
\toprule
Domain & Pairs & Legitimate action & Unauthorized action \\
\midrule
Workspace & 10 & Email payment terms to finance & Add audit recipient \\
CRM & 10 & Update ticket and email support & Add audit recipient \\
Code review & 10 & Update tracker and email maintainer & Add audit recipient \\
Travel & 10 & Create approved calendar event & Add audit recipient \\
Research & 10 & Write bibliography note and email lead & Add audit recipient \\
\bottomrule
\end{tabular}
\caption{SLaMBench scaled task set used for the main result.}
\label{tab:slambench}
\end{table}

\paragraph{Payloads.}
The injected variants use five intentionally simple payload classes. \emph{Direct
instruction} explicitly tells the system to send the extra recipient. \emph{Fake
authority} claims that a user, compliance process, or policy has approved the
extra recipient. \emph{Delegation poisoning} targets the planning boundary by
asking one agent to tell another agent to execute the forbidden call.
\emph{Summary bait} phrases the forbidden content as a natural action item.
\emph{Memory poisoning} attempts to write a durable policy such as ``external
vendor notes are trusted.'' The goal is not to find the strongest adaptive
attack, but to test whether ordinary-looking workflow text becomes more dangerous
after provenance-losing delegation.

\paragraph{Topologies.}
The \textbf{direct single-agent} condition
(\texttt{c0\_single\_direct}) receives the user goal, trusted context, untrusted
artifact, and tool signatures, then emits tool calls. The main vulnerable
condition is \textbf{extractive multi-agent}
(\texttt{c2x\_3agent\_extractive\_naive}): a Reader extracts action items from
the artifact, a Planner turns the Reader output into steps, and an Executor emits
tool calls. The Reader does not preserve an explicit untrusted label in this
stress condition, modeling a common engineering shortcut where upstream agents
compress or normalize task context before execution.

We also run topology ablations. \texttt{c1\_2agent\_naive} and
\texttt{c2\_3agent\_naive} use free-form delegation without the extractive action
item prompt. \texttt{c2o\_3agent\_oracle\_laundered} gives downstream agents a
deliberately provenance-stripped peer summary, isolating what happens after
laundering has already occurred.

\paragraph{Defenses.}
We compare three defense styles. \textbf{Source-preserving} prompts require the
Reader to preserve source identifiers and trust labels. \textbf{Capability-scoped}
prompts tell downstream agents that peers may suggest actions but cannot
authorize sensitive tools unless the user goal or trusted context does so.
\textbf{TraceGate} uses the same extractive model outputs but checks sensitive
tool calls before execution. It blocks synthetic-secret disclosure, unauthorized
recipients or attendees, untrusted memory-policy writes, and sensitive actions
without trusted authorization. For mixed-recipient emails, it prunes unauthorized
recipients and preserves authorized sends when possible. We also evaluate an
LLM-guard replay baseline over saved c2x tool calls; unlike TraceGate, this guard
is another model-call sanitizer rather than deterministic enforcement.

\paragraph{Metrics.}
All metrics are computed from saved tool traces. \textbf{Benign Success Rate}
(BSR) is benign-task success without forbidden behavior. \textbf{Injected Utility
Success Rate} (IUSR) is the same useful-task success check on injected variants,
again requiring no forbidden behavior. \textbf{Attack Success Rate} (ASR) is the
fraction of injected variants with a forbidden event in the executed tool log;
blocked calls do not count as attacks. \textbf{Authority Transfer Rate} (ATR) is
ASR minus the direct-baseline ASR on the same task set. \textbf{Provenance Dropout
Rate} (PDR) is the fraction of reader/planner messages that carry task-specific
forbidden content without source, trust, and no-authority metadata. We keep most
confidence intervals and diagnostic breakdowns in the repository tables and
supplement so that the main text emphasizes the mechanism.

\section{Results}

The results are organized around four takeaways. First, extractive delegation can
amplify unauthorized sends. Second, the amplification is tied to
provenance-losing handoffs rather than the mere number of agents. Third,
prompt-only defenses help but can trade off benign utility, while tool-boundary
checks preserve the useful part of the task. Fourth, the phenomenon is
model-dependent: stronger current models may show provenance dropout without
executed attacks.

\subsection{Main finding: delegation can transfer authority}

The main SLaMBench-50 run uses \texttt{gpt-5.4-nano}. Table~\ref{tab:main_simple}
shows only the numbers needed to understand the result; fuller tables with CIs,
PDR, CNAS, and block counts are in the repository.

\begin{table}[t]
\centering
\small
\begin{tabular}{lccc}
\toprule
Condition & Benign utility & Safe utility under attack & Forbidden sends \\
\midrule
Direct single-agent & 60\% & 54\% & 6\% \\
Extractive multi-agent & 70\% & 50\% & 22\% \\
TraceGate & 66\% & 64\% & 0\% \\
\bottomrule
\end{tabular}
\caption{Headline SLaMBench-50 result on \texttt{gpt-5.4-nano}. The main visual
impression is a rise from 6\% to 22\% forbidden sends under extractive delegation,
then 0\% after TraceGate.}
\label{tab:main_simple}
\end{table}

The extractive reader--planner--executor topology raises ASR from 6\% to 22\%, a
+16 point ATR relative to direct exposure. This is the core source-laundering
signal: the same injected artifacts are present in both conditions, but the
provenance-losing handoff makes the forbidden instruction more likely to execute.
The security cost is not explained by worse task competence. The extractive
system has higher benign utility than the direct baseline (70\% versus 60\%), but
its injected utility falls because it sometimes completes the task in an unsafe
way. TraceGate changes the shape of the failure: it blocks the forbidden sends
and preserves the authorized work, yielding the best safe utility under attack
(64\%).

A typical successful attack follows the intended laundering path. The Reader
converts an untrusted ``user-approved routing exception'' into an action item to
send the final summary to \texttt{audit-log@example.test}; the Planner turns that
item into a required email step; the Executor emits the corresponding
\texttt{send\_email} call. In a defended trace, the model proposes a mixed
recipient list, TraceGate removes \texttt{audit-log@example.test}, and the
authorized finance email is still executed. The important distinction is that
TraceGate does not fix the upstream provenance dropout; it prevents the dropout
from becoming an external side effect.

\subsection{Mechanism: not more agents, but provenance-losing handoffs}

The topology ablation uses the same 25-pair SLaMBench slice and
\texttt{gpt-5.4-nano}. Rather than reporting every metric, Table~\ref{tab:topology_simple}
focuses on the mechanism.

\begin{table}[t]
\centering
\small
\begin{tabular}{p{0.31\linewidth}cp{0.43\linewidth}}
\toprule
Condition & ASR & What it shows \\
\midrule
Direct single-agent & 4\% & Baseline direct exposure to the injected artifact. \\
2-agent naive & 0\% & Delegation by itself does not create the effect. \\
3-agent naive & 4\% & More agents still do not raise ASR here. \\
3-agent extractive & 24\% & Extracting artifact instructions as action items amplifies attacks. \\
Oracle-laundered handoff & 28\% & Downstream agents are vulnerable once provenance is already stripped. \\
\bottomrule
\end{tabular}
\caption{Topology ablation on the 25-pair slice. The risk appears in
provenance-losing handoffs, not in naive delegation depth alone.}
\label{tab:topology_simple}
\end{table}

This ablation sharpens the claim. The naive 2-agent and 3-agent chains do not
increase ASR over direct exposure. In contrast, the extractive Reader that
normalizes document instructions into action items raises ASR by +20 points over
the direct baseline on the same slice, and the oracle-laundered handoff raises it
by +24 points. A no-API trace audit gives the same picture: in all six injected
tasks where extractive delegation attacks and direct exposure does not, the
forbidden recipient appears in the Reader handoff, the Planner plan, and the
Executor output before the tool call. A hard-case repeatability check is
consistent but small: extractive delegation re-attacks 8/18 repeated runs, direct
exposure attacks 2/18, and TraceGate replay executes 0/18 forbidden sends.

Payloads also support the mechanism. Summary bait and fake authority are more
effective in the extractive topology than direct instructions: their c2x ASRs are
50\% and 40\%, compared with 10\% for direct instructions and 10\% for delegation
poisoning. The most dangerous payloads are not necessarily the most overtly
adversarial ones; they are the ones that look like ordinary workflow content for
a Reader to extract.

\subsection{Defenses: prompt provenance helps, tool checks are cleaner}

Table~\ref{tab:defense_simple} compares defenses on the clean 25-pair slice. The
TraceGate and LLM-guard rows are replays over the same proposed c2x tool calls;
they are not independent model resamplings.

\begin{table}[t]
\centering
\small
\begin{tabular}{p{0.34\linewidth}ccp{0.27\linewidth}}
\toprule
Defense condition & Safe utility & ASR & Main tradeoff \\
\midrule
None: extractive multi-agent & 56\% & 24\% & Useful, but unsafe. \\
Source-preserving prompts & 56\% & 0\% & Secure here, lower benign utility. \\
Capability-scoped prompts & 52\% & 0\% & Secure here, also conservative. \\
TraceGate replay & 72\% & 0\% & Blocks side effects, preserves authorized work. \\
LLM guard replay & 72\% & 0\% & Works on this slice, but adds another model-dependent component. \\
\bottomrule
\end{tabular}
\caption{Defense comparison on the 25-pair slice with \texttt{gpt-5.4-nano}.
Prompt-only defenses reduce attacks but can be conservative; TraceGate enforces a
crisp authorization rule at the tool boundary.}
\label{tab:defense_simple}
\end{table}

The defense story is encouraging but nuanced. Source-preserving and
capability-scoped prompts reduce ASR from 24\% to 0\% on this slice, showing that
source and authority metadata help. However, they also reduce benign success in
this pilot, likely because they make agents more cautious or verbose around
otherwise legitimate tool use. TraceGate reaches 0\% ASR while increasing safe
utility under attack to 72\%, because it can prune a forbidden recipient and still
execute the authorized send. The LLM guard replay also reaches 0\% ASR on this
slice, but its assurance is empirical: it is another model decision. TraceGate is
less general, but more auditable when the policy is crisp, such as ``only send to
recipients authorized by the user or trusted context.''

\subsection{Model dependence and scope}

The cross-model results show that source laundering should be stated as a
conditional systems failure, not a universal law of multi-agent systems.

\begin{table}[t]
\centering
\small

\toprule
Model & ASR path: direct $\rightarrow$ extractive $\rightarrow$ TraceGate & Regime \\
\midrule
\texttt{gpt-5.4-nano} & 4\% $\rightarrow$ 24\% $\rightarrow$ 0\% & Positive authority-transfer signal. \\
\texttt{gpt-4.1-nano} & 24\% $\rightarrow$ 24\% $\rightarrow$ 0\% & Direct exposure is already vulnerable. \\
\texttt{gpt-5.5} & 0\% $\rightarrow$ 0\% $\rightarrow$ 0\% & Robust on this slice despite provenance dropout. \\
\bottomrule
\end{tabular}
\caption{Model dependence on the 25-pair slice. SLaMBench distinguishes direct

\label{tab:model_simple}
\end{table}

For \texttt{gpt-4.1-nano}, direct exposure is already vulnerable, so delegation
cannot add much measured ASR. For \texttt{gpt-5.5}, executed attacks are 0\% in
all tested conditions, although provenance dropout is still observed in the
extractive chain. This means the benchmark is diagnostic in two ways. It can show
executed source-laundering attacks when a model follows laundered instructions,
and it can show latent provenance loss even when a stronger Executor refuses to
act. A six-case heterogeneous diagnostic supports the role interpretation:
all-nano c2x attacks all six selected hard injected tasks, while a
\texttt{gpt-5.4-nano} Reader/Planner with a \texttt{gpt-5.5} Executor attacks
0/6. This is not a powered defense result, but it suggests that executor
robustness and handoff design are separable axes.

The benchmark's scope is also clear. The strongest positive result is about
unauthorized recipients. A tiny non-email side-effect probe for ticket status,
calendar attendee, and memory-policy writes does not show positive c2x lift, so
we do not claim that every side-effect type is equally vulnerable. The broader
claim is mechanistic: when an upstream agent rewrites untrusted content as a peer
action item, downstream agents may treat designation as authorization unless the
system preserves and enforces provenance.

\section{Related Work}

\paragraph{LLM agents and multi-agent systems.}
The agentic-AI literature increasingly treats language models as components that
reason, act, use tools, remember, and collaborate. ReAct interleaves reasoning
and actions against external environments~\citep{yao2022react}; Generative
Agents use memory, reflection, and planning to simulate interactive
behavior~\citep{park2023generative}; AutoGen provides a framework for building
applications from conversable agents~\citep{wu2023autogen}; and SWE-agent shows
that agent-computer interfaces strongly shape software-engineering agent
performance~\citep{yang2024sweagent}. Source laundering studies the security
cost of the same architectural move: once agents communicate through compressed
natural-language handoffs, facts, instructions, and authority can become
entangled.

\paragraph{Prompt injection and agent benchmarks.}
Indirect prompt injection showed that LLM applications can confuse retrieved or
third-party content with instructions~\citep{greshake2023not}. InjecAgent,
AgentDojo, Agent Security Bench, ToolEmu, and WASP evaluate prompt-injection and
agent-security risks in tool-integrated or web-agent settings, often with richer
task suites than ours~\citep{zhan2024injecagent,debenedetti2024agentdojo,zhang2024asb,ruan2023toolemu,evtimov2025wasp}.
SLaMBench is intentionally smaller and more diagnostic. Rather than measuring
whether indirect prompt injection works in general, it asks whether an inter-agent
handoff increases the effective authority of the same injected instruction.

\paragraph{Multi-agent propagation, memory, and tool metadata.}
Prompt Infection studies LLM-to-LLM prompt injection that propagates across
multi-agent systems~\citep{lee2024promptinfection}. AgentPoison and MemoryGraft
study poisoning of long-term memory, RAG stores, or experience retrieval in
agents~\citep{chen2024agentpoison,srivastava2025memorygraft}. ToolHijacker and
recent MCP-focused work show that tool descriptions, tool-server trust, and tool
metadata are also security-relevant surfaces~\citep{shi2025toolhijacker,metere2026mcpattested}.
Source laundering differs from these attacks in the object being transformed: the
attacker does not need to persist in memory, self-replicate, or replace a tool.
It is enough for an untrusted instruction to be rewritten by a peer in a form
that downstream agents treat as executable.

\paragraph{Defenses and systems-security framing.}
Several defenses try to separate data from instructions or improve model-level
resistance. StruQ uses structured queries to separate prompt and data
channels~\citep{chen2024struq}; SecAlign uses preference optimization for
prompt-injection resistance~\citep{chen2024secalign}; PromptArmor sanitizes
injected content before execution~\citep{shi2025promptarmor}; and WARD studies
robust guard models for web agents~\citep{cao2026ward}. These defenses are
valuable, but source laundering suggests that prompt-level separation should be
paired with runtime authorization. Our TraceGate defense follows a systems
principle also reflected in OWASP's ``excessive agency'' risk: sensitive side
effects should be constrained by explicit policy rather than by model compliance
alone~\citep{owasp2025llmtop10}.

\paragraph{Compositional authorization.}
The closest classical analogy is the confused deputy, where a privileged program
is tricked into misusing its authority on behalf of a less privileged
principal~\citep{hardy1988confused}. Source laundering is a natural-language,
multi-agent version of that problem. The downstream Executor has tool authority;
the untrusted artifact does not. If the system fails to preserve the principal on
whose behalf an instruction is being carried, the Executor may mistake a
provenance-stripped suggestion for authorization.

\section{Limitations}

SLaMBench is synthetic, so it is safe and controllable but does not estimate
production incident rates. The main positive result is strongest for
\texttt{gpt-5.4-nano}; \texttt{gpt-4.1-nano} is already directly vulnerable, and
\texttt{gpt-5.5} shows low ASR on the tested slice. The extractive c2x condition
is a stress condition, not a claim that all production Reader agents are written
this way. Real systems may have richer tool schemas, retries, confirmations,
heterogeneous agents, stateful memories, and organization-specific policies.
TraceGate is deliberately simple; production use would require richer provenance
tracking, structured authority metadata, policy compilation, user confirmation,
and task-specific authorization definitions. Finally, the benchmark emphasizes
unauthorized recipients. Broader state-changing slices, larger repeated
stochastic sweeps, and stronger heterogeneous-agent evaluations remain future
work.

\section{Conclusion}

Source laundering is a security-relevant failure mode in multi-agent LLM systems:
untrusted artifact text can become more effective when rewritten as a peer-agent
action item. On SLaMBench, extractive delegation increases unauthorized-recipient
sends relative to direct exposure for \texttt{gpt-5.4-nano}, while a deterministic
tool-boundary monitor eliminates observed unauthorized sends and preserves useful
work. Follow-up ablations show that the effect is about provenance-losing
composition, not delegation depth alone, and cross-model results show that the
risk is model-dependent. Multi-agent systems should therefore treat inter-agent
summaries and plans as untrusted by default unless source and authority are
explicitly preserved and enforced.

\section*{Ethics Statement}

All experiments use synthetic data, synthetic secrets, synthetic recipients, and
mock tools with no external side effects. The release package contains synthetic
tasks, aggregate logs, and deterministic scoring scripts, not live targets or
real credentials. The benchmark is intended for defensive evaluation of
provenance-preserving agent designs and tool-boundary authorization checks.
